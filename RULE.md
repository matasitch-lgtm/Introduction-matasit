# Rule: DevPool 2569 Mini Build Challenge

> เอกสารนี้สรุปจากหน้า [ภารกิจรอบคัดเลือก DevPool 2569](https://cumbersome-barracuda-6ba.notion.site/DevPool-2569-3828722d2e9080feb599d4330556e533?pvs=143) ณ วันที่ 21 มิถุนายน 2569 โดยยึดประกาศต้นฉบับเป็นหลัก หากโครงการเปลี่ยนแปลงรายละเอียดภายหลัง

## เป้าหมายของงาน

สร้างหน้าเว็บแนะนำตัว 1 หน้า ซึ่งเปิดใช้งานได้จริง มีการโต้ตอบกับผู้ใช้อย่างน้อย 1 จุด และส่งโค้ดผ่าน GitHub

โครงการไม่ได้เน้นว่างานต้องสวยหรือมีฟีเจอร์ครบ แต่ต้องการดูว่า:

- ผู้สมัครตั้งใจทำงานจริง
- ผู้สมัครเข้าใจและอธิบายโค้ดที่ส่งได้
- ผลงานเป็นเรื่องราวของผู้สมัครเอง

## ข้อกำหนดขั้นต่ำ (ต้องมีครบ)

- มีไฟล์ `index.html` ที่เปิดด้วยเบราว์เซอร์แล้วแสดงผลได้
- มีเนื้อหาแนะนำตัว ได้แก่
  - ชื่อ
  - สังกัดหรืองานที่ทำอยู่ใน กฟภ.
  - เหตุผลที่อยากเข้า DevPool
- ใช้ CSS จัด layout สี และตัวอักษรอย่างตั้งใจ ไม่ใช่ HTML เปล่า
- ใช้ JavaScript ทำ interaction อย่างน้อย 1 จุด โดยมีการใช้ DOM ร่วมกับ Event
- นำโค้ดขึ้น public repository บน GitHub

ตัวอย่าง interaction ที่เลือกทำได้ เช่น สลับ Light/Dark mode, แสดงหรือซ่อนข้อมูล, แสดงข้อความตามที่พิมพ์, นับจำนวนคลิก หรือเปลี่ยนข้อความ/รูปเมื่อ hover หรือคลิก

## ส่วนเสริม (ไม่บังคับ)

- ดึงข้อมูลจาก Public API ฟรีด้วย `fetch` แล้วนำมาแสดง เช่น คำคมหรือสภาพอากาศ
- ทำ responsive ให้ใช้งานได้ดีทั้งมือถือและจอใหญ่
- Deploy ผ่าน GitHub Pages หรือ Vercel และแนบลิงก์เว็บจริง

ทำเฉพาะข้อกำหนดขั้นต่ำก็ผ่านได้ หากส่งครบ มาสัมภาษณ์ และอธิบายโค้ดได้

## กติกาการใช้ AI

- ใช้ AI ได้ทุกตัวและทุกขั้นตอน รวมถึง Vibe Coding ไม่ถือว่าโกง
- ต้องอธิบายได้ว่าโค้ดทุกส่วนทำอะไรและทำไมจึงอยู่ตรงนั้น
- ควรอ่าน ทดลองแก้ และลองลบโค้ดเพื่อดูผลจนเข้าใจ ก่อนส่งงาน
- ต้องบอกตามจริงว่า AI ช่วยส่วนใด และอธิบายสิ่งที่ได้เรียนรู้จากส่วนนั้น

หลักสำคัญคือ โครงการไม่ได้วัดว่าใครเป็นคนพิมพ์โค้ด แต่วัดว่าผู้สมัครเข้าใจสิ่งที่ส่งหรือไม่

## วิธีส่งงาน

1. สร้าง public repository บน GitHub
2. นำโค้ดขึ้น repository และควรทยอย commit หลายครั้งระหว่างพัฒนา
3. ส่งลิงก์ Git repository ทางอีเมล `nattchanatip.thi@pea.co.th` ภายในเวลา 23:59 น. วันที่ 29 มิถุนายน 2569
4. หาก Deploy เว็บจริง ให้แนบลิงก์เว็บด้วย (ไม่บังคับ)

## เกณฑ์ผ่านและไม่ผ่าน

ผ่านเมื่อ:

- ส่งงานครบตามข้อกำหนดขั้นต่ำ
- มาสัมภาษณ์ตามนัด
- อธิบายโค้ดของตัวเองได้

ไม่ผ่านเมื่อ:

- ไม่ส่งงานหรือส่งไม่ครบข้อกำหนดขั้นต่ำ
- ไม่มาสัมภาษณ์
- อธิบายโค้ดของตัวเองไม่ได้เลยแม้แต่จุดเดียว

## สิ่งที่จะเจอในการสัมภาษณ์

ช่วงสัมภาษณ์กำหนดไว้วันที่ 1–5 กรกฎาคม 2569 โดยวันเวลาอาจเปลี่ยนแปลงตามความเหมาะสม คำถามหลักมีแนวโน้มเป็น:

1. โค้ด JavaScript บรรทัดนี้ทำอะไร
2. ถ้าจะเพิ่มฟีเจอร์ เช่น เพิ่มปุ่ม ต้องแก้ส่วนใด
3. AI ช่วยตรงไหน และผู้สมัครเข้าใจส่วนนั้นอย่างไร

## ขอบเขตของโจทย์

- รอบนี้เป็น Frontend หน้าเดียว
- ยังไม่ต้องทำ Backend หรือ Database
- ควรเรียน HTML, CSS, JavaScript, DOM, Event และ Git ให้เข้าใจ โดย DOM และ Event สำคัญโดยตรงกับ interaction

## สิ่งที่ควรเลี่ยง

- ลอก template ทั้งหน้า หรือให้ AI สร้างทุกอย่างโดยไม่อ่านและไม่เข้าใจ
- ทำฟีเจอร์มากเกินไปจนไม่สามารถอธิบายโค้ดของตัวเองได้
- รอ commit ครั้งเดียวตอนงานเสร็จ
- ส่งงานแล้วไม่มาสัมภาษณ์ หรือไม่เตรียมอธิบายโค้ด

## กติกาการทำงานร่วมกับ Codex ในโปรเจกต์นี้

- Codex เป็นผู้ช่วยสอนและให้คำแนะนำเท่านั้น
- Codex ห้ามแก้ สร้าง หรือลบโค้ดของผู้ใช้เอง เว้นแต่ผู้ใช้จะเปลี่ยนคำสั่งและอนุญาตอย่างชัดเจนในภายหลัง
- เมื่อผู้ใช้ขอความช่วยเหลือ Codex ควรอธิบายแนวคิด ชี้ตำแหน่งที่ควรตรวจ และเสนอขั้นตอนให้ผู้ใช้ลงมือพิมพ์หรือแก้เอง
- หากยกตัวอย่างโค้ด ต้องอธิบายหน้าที่ของแต่ละส่วนและเหตุผลที่ใช้ เพื่อให้ผู้ใช้สามารถตอบคำถามสัมภาษณ์ได้
- Codex ควรถามให้ผู้ใช้ลองอธิบายโค้ดกลับ หรือเสนอแบบฝึกทดลองแก้/ลบทีละจุดเมื่อเหมาะสม
- Codex ห้ามทำงานทั้งหมดแทนผู้ใช้ เพราะเป้าหมายคือให้ผู้ใช้เข้าใจและนำเสนอผลงานของตนเองได้
